The usual approach is to form a list of lists in order of time, called a ‘time wheel’, as shown in Fig. 6.9. 

. 
The first list is a set of pointers to the event memory. 

. 
There is notionally one pointer for each time interval from ‘T+1’. This could be a very large number. 

. 
Thus, one might limit the number of pointers to 1024, and re-use them at every 1024 time steps. The list pointers is thus a circular list, and is called a time WHEEL or chronwheel. The wheel itself is an array of pointers. 



. 
Each pointer in the time wheel indicates the address of a list of events that has been scheduled for that time. 

. 
Each list must be long enough to hold the maximum number of events for any time interval. 

. 
Furthermore, the lists for the times furthest ahead are likely to have relatively few events. 

. 
This is a very poor use of storage. 


. 
An alternative is to use a linked list for each time slot in the time wheel. It is only necessary to provide sufficient storage for the sum of all the event lists, and there is much less wastage. 

. 
The event memory consists of a series of records as shown in Fig. 6.10. each record holds the predicted value of a signal, the fan-out index(FOI), and a pointer. 



. 
The new event is added to the list (which) indicated by the pointer. 

. 
If the pointer is null, the time wheel and event memory are shown as Fig. 6.11(a). 

. 
Fig. 6.11(b) illustrates the case where the pointer in the time wheel is 31. The pointer of record 31 is null as this is the only event in this list. 

. 
There are now two events in the set for this time, the new event having been added to the front of the list. The list is indicated by the time wheel value of 137. 

. 
The order of events in a particular time slot is not important.

Notice that the events are withdrawn from the event list in reverse order to that in which they were placed there. It is axiomatic that the order of processing of all events in a given list, which take place at the same time, is unimportant. 

. If the order is important then some ‘splitting’ of time slots is required. The function of deltas in VHDL is to force these extra time slots.

To determine which elements to evaluate in the new time slot, it is necessary to know which circuits are driven. Each output signal has an entry in the fan-out memory of Fig. 6.7, which is a table containing details of where the output is in the network. 

. Fig. 6.12 illustrates this. A NAND gate drives two loads, a flip-flop and a logic block called FRED. Suppose the record in the network representing the NAND gate begins at address 258. 

. 
Conventional wisdom suggests that each output drives 2.5 inputs on average, so for each event, there are 3.5 sets of data out, that is for the output and 2.5 inputs(of the driven elements). 

. 
The state table(Fig. 6.7) holds the current state of all outputs, and is a copy of those in the network memory. 

. Referring back to the example of Fig. 6.8, at time 12 the state table contains ‘1’ for signal C, but the predicted value is ‘0’. At time 14, the values for the state table and the predicted are ‘0’ and ‘1’, respectively. 

.
