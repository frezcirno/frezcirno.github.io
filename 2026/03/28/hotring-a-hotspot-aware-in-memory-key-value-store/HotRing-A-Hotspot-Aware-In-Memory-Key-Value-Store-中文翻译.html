#摘要

内存键值存储（KVS）被广泛用于缓存热点数据，以缓解基于磁盘的存储系统或分布式系统中的热点问题。然而，内存 KVS 自身内部的热点问题却长期被忽视。随着近年来热点现象愈发严重，现有 KVS 由于缺乏热点感知能力，在高度偏斜的负载下往往表现不佳，甚至不够可靠。

本文探索了面向内存 KVS 索引结构的热点感知设计。我们首先分析了理想热点感知索引所能带来的潜在收益，并讨论了有效利用热点感知时面临的挑战，即：热点迁移与并发访问。基于这些观察，我们提出了一种新的热点感知 KVS——HotRing，它针对“大量并发请求集中访问少量条目”的场景进行了优化。

HotRing 基于一种有序环哈希索引结构：通过将桶头指针移动到更靠近热点项的位置，使热点访问更快；同时，它还采用一种轻量级机制在运行时检测热点迁移。HotRing 在整体设计中系统性地采用了无锁结构，不仅覆盖常规操作（如读、更新），也覆盖了 HotRing 特有操作（如热点迁移检测、头指针移动以及有序环 rehash），从而使大规模并发请求更好地利用多核架构。大量实验表明，在高度偏斜的工作负载下，HotRing 相比其他内存 KVS 最多可取得 2.58× 的性能提升。

#1. 引言

内存键值存储（KVS）是存储基础设施中的关键组成部分，例如数据库和文件系统常使用它在内存中缓存高频访问数据，以获得更快的访问速度。KVS 能够提升这些系统的性能与可扩展性，因为在很多实际场景中，系统每秒需要处理数十亿次请求。许多最先进的 KVS，如 Memcached、Redis 以及它们的各种变体，已经在 Facebook、Amazon、Twitter 和 LinkedIn 等企业的生产环境中广泛部署。

所谓热点问题，是指在高度偏斜的负载下，只有很小一部分数据项被频繁访问。这是现实系统中的常见现象，学术界也已对其进行过广泛研究。为了解决集群范围的热点问题，人们提出了很多方法，例如一致性哈希、数据迁移与前端缓存等。与此同时，单节点层面的热点问题也在其他层面得到了较好处理。例如，计算机体系结构通过分层存储布局（磁盘、内存、CPU Cache）把高频访问数据缓存在低延迟介质中；许多存储系统，如 LevelDB 和 RocksDB，也会使用内存 KVS 管理热点数据。

然而，内存 KVS 内部的热点问题通常被忽略。作者从阿里巴巴的生产环境中采集了若干内存 KVS 的访问分布，观察到在日常场景中约有 50% 的访问落在 1% 的条目上，而在极端场景中，这一比例可高达 90%。这表明在互联网时代，热点问题已经异常严重。

出现这种现象有几个原因。首先，在线应用中的活跃用户规模持续扩大，某些实时事件（如大促、突发新闻）会在很短时间内把数十亿次访问集中到少量数据上，因此必须极快地处理这些热点。已有报告指出：Amazon 页面加载每延迟 0.1 秒就会损失约 1% 销售额；Google 搜索结果如果增加 0.5 秒加载时间，流量可能下降 20%。其次，底层基础设施愈发复杂，链路中某个细小错误（例如软件 bug 或配置失误）也可能导致某个条目被异常重复访问，例如不断读取同一数据并返回错误消息。理想情况下，这类不可预测的热点不应拖垮整个系统。因此，在热点存在时依然让 KVS 保持高性能和高可靠性非常重要。

实现 KVS 的索引结构有很多种，例如跳表、平衡树 / Trie 树（如 Masstree）以及哈希。由于查找速度快，哈希通常是最流行的方案。然而，现有大多数方法都不感知热点：它们用同一种策略对待所有条目，因此访问热点项与访问冷数据项在逻辑上需要相同数量的内存访问。从理论分析（详见第 2.2 节）可以发现，当前哈希索引中访问热点所需的努力，远高于理想热点感知方案。现有机制虽然能降低一些内存访问次数，但效果有限。例如，CPU Cache 能加速热点访问，但容量通常只有约 32MB；rehash 能缩短冲突链，但会显著增加内存占用。这说明，在高度偏斜负载下，针对热点访问仍然存在进一步优化空间。

本文提出 HotRing：一种热点感知的内存 KVS，它使用一种经过专门优化的哈希索引，以支持大量并发请求访问少数热点项。核心思想是：查找一个条目所需的内存访问次数，应该与该条目的热度负相关，即越热的数据应越快读到。

为了实现这一目标，必须解决两个关键挑战：

  1. 热点迁移（hotspot shift）：热点集合会随着时间变化，因此系统需要及时识别并适配新的热点。
  2. 并发访问（concurrent access）：热点项天然会被大量线程同时访问，系统必须在这些热点上维持高并发能力。

针对热点迁移问题，作者将传统哈希索引中的冲突链替换为一种有序环结构，使桶头指针可以随着热点变化直接重新指向更热点的位置，而不破坏正确性。同时，系统还采用轻量机制在运行时探测热点迁移。针对并发访问问题，HotRing 采用了受已有无锁结构启发的设计，并将其扩展到 HotRing 所需的全部操作，包括热点迁移检测、头指针移动和有序环 rehash。

作者在模拟真实工作负载的基准测试上进行了大量实验，将 HotRing 与基于无锁链表的哈希结构及其他基线系统进行了比较。结果显示，在极端偏斜负载下，HotRing 每秒最多可处理 5.65 亿次读请求，相较其他系统最高可获得 2.58× 提升；对于原地更新与 RCU 更新，分别取得 2.17× 和 1.32× 的改进。这表明，HotRing 是提升单机热点处理能力、增强内存 KVS 性能与可靠性的有效结构。

#贡献总结

本文的主要贡献如下：

  • 指出了现有内存索引中的热点问题，并展示了热点感知设计在提升热点访问性能方面的巨大潜力。
  • 提出了 HotRing——一种有序环哈希结构，作为热点感知设计的首次系统性尝试。它通过将头指针移动到更靠近热点的位置，提升热点访问速度；同时采用轻量策略在运行时检测热点迁移。
  • 将 HotRing 设计为无锁结构，以支撑海量并发访问；特别是从零开始设计了 HotRing 特有操作，包括热点迁移检测、头指针移动和有序环 rehash。
  • 在基于真实工作负载特征的实验中验证了该方法的有效性。结果表明，当访问分布高度偏斜时，HotRing 显著优于其他 KVS。

#2. 背景与动机

本节首先介绍现有 KVS 中常见的哈希索引以及其热点问题；随后从理论上说明理想热点感知哈希所能带来的潜在收益；最后讨论在实际索引中利用热点感知时面临的挑战以及相应设计原则。

#2.1 哈希索引与热点问题

哈希索引是 KVS 中最常见的内存结构，尤其适用于上层应用不需要范围查询的场景。传统哈希索引通常由一个全局哈希表与每个桶对应的一条冲突链组成。访问某个 key 时，先计算其哈希值 h，定位到相应桶头，然后沿冲突链逐个比较，直到找到目标 key 或到达链尾（即 key 不存在）。

一个 n 位哈希值通常可分为两部分：一部分用于定位哈希表桶（例如 k 位），另一部分作为 tag（例如 n-k 位）。tag 可以存放在条目中，以避免直接比较长 key。

问题在于，传统哈希索引并不感知热点。热点项可能均匀散落在冲突链中。例如，一个热点项如果靠近链尾，那么它相比链前部的数据需要更多次内存访问。在高度偏斜的负载中，哪怕热点项访问成本只是略微上升，也可能显著拖累整体性能。

已有方法虽然能在一定程度上降低热点项访问成本，但效果有限：

  1. CPU Cache 可以缓存热点数据块，但容量通常只有约 32MB，而整机内存往往超过 256GB，可缓存比例极低，远小于实际热点比例。
  2. 扩大哈希表 / rehash 可以缩短冲突链，但代价是显著增大内存占用；而且哈希表越大，再继续翻倍扩容带来的收益会递减。

因此，现有方法都只是部分缓解了热点问题，并没有真正从索引结构本身做“热点感知”优化。

#2.2 热点感知带来的潜在收益

由于热点问题越来越严重，一个自然问题是：如果哈希索引能够感知热点，究竟有多大收益？

在传统基于冲突链的哈希索引中，热点项随机分布在链上，因此热点与冷数据在访问代价上没有区别。假设共有 N 个条目，哈希表有 B 个桶，则平均每条冲突链长度为：

L = N / B

检索一个条目时，期望内存访问次数为：

E_chain = 1 + L/2 = 1 + N/(2B)

其中前面的 1 表示访问哈希表桶本身。

理想的热点感知哈希应该让检索代价与热度负相关，也就是最热的数据需要最少的访问次数。作者使用 Zipf 分布建模条目的访问热度，令第 x 热条目的访问频率为：

f(x) = (1/x^q) / Σ(1/n^q)

其中 q 是偏斜参数。

为了便于分析，作者假设热点在不同桶中均匀分布，即每条链上都有一个最热区间中的条目、一个次热区间中的条目，依此类推。如果能够在每条链中按访问频率降序排列条目，那么理想热点感知哈希的期望访问代价 E_ideal 会显著低于传统链式哈希。

作者对传统哈希与理想热点感知哈希进行了理论计算。结果显示：随着冲突链长度增长，热点感知哈希的访问效率明显优于传统哈希。这说明，把“热点意识”纳入哈希索引设计，是一个极具潜力的优化方向。

#2.3 挑战与设计原则

尽管热点感知是有益的，但在实际设计中仍面临两个核心挑战：

  • 热点迁移：真实应用中的访问模式随时间变化，不可能始终把所有条目完美地按最新热度排序，因此需要一种轻量方式跟踪热点变化。
  • 并发访问：热点项会被大量并发请求同时访问，因此必须保证读写操作都具备高并发能力。

针对热点迁移问题，HotRing 的设计原则是：不去频繁重排链中所有条目，而是移动头指针。例如，把头指针移动到当前最热项，或一个全局更优的位置。为保证无论头指针如何移动，都仍然能够访问桶内所有条目，作者用有序环替代传统冲突链。虽然这种设计达不到第 2.2 节中理想方案的最优状态，但实验表明它已经足够有效且执行速度快。此外，HotRing 使用两种轻量策略在运行时探测热点迁移。

针对并发访问问题，经典思路是采用无锁结构，避免昂贵的锁与同步开销。已有研究证明，无锁设计可以显著提升系统吞吐量。HotRing 在设计中借鉴了既有无锁链表工作，并在此基础上扩展支持全部 HotRing 所需操作，包括热点迁移检测、头指针移动以及 rehash。

#3. HotRing 的设计

本节详细介绍 HotRing 的设计，包括热点感知索引结构、热点迁移检测策略，以及各种无锁操作（读 / 写、插入 / 删除、头指针移动与 rehash）。

#3.1 有序环哈希索引

HotRing 对传统哈希索引中的冲突链进行了改造：将链尾重新连回链首，从而形成一个冲突环。这样一来，哈希表中的头指针就不再必须指向链首，而是可以指向环中的任意条目。如果一个环中只有一个条目，那么它的 next 指针就指向自己。

这种设计允许系统根据热点变化移动头指针，把它移到更接近热点的位置。但由此也带来一个问题：在环结构中，如果目标条目不存在，简单遍历会陷入无限循环。因此必须能安全地判断何时终止查找。

为了解决这个问题，HotRing 将每个环中的条目按 (tag, key) 的字典序排序。这样一来，查找某个目标条目时：

  • 如果当前条目的 (tag, key) 与目标完全相等，则说明命中；
  • 如果已经遇到一对“前一个条目小于目标、当前条目大于目标”的相邻条目，则可确定目标不存在；
  • 或者在环绕过程中，根据有序性判定目标已经不可能出现。

这样，HotRing 可以在环上安全终止查找，而不必回到固定链尾。更重要的是：与传统链式哈希不同，在 HotRing 中，对于未命中查找，平均只需要检查约 (n/2)+1 个条目，而不是全部 n 个。

#3.2 热点迁移识别

在有序环哈希索引中，查找过程已经能够判断命中与未命中。接下来要解决的是：如何识别热点，并在热点迁移时调整头指针。

由于哈希值通常分布均匀，因此热点条目也会大致均匀分布到不同桶中。实践中，每个桶里的冲突条目数相对较少（例如 5～10 个），因此在 10%～20% 热点比例下，每个环里通常只有一个主要热点。将头指针直接指向这个热点项，既能提升性能，又不需要真的重排所有数据。

识别热点需要权衡两个指标：

  • 识别准确率：识别出的热点是否真的是热点；
  • 反应延迟：从热点真正发生变化，到系统识别出新热点，中间经历多长时间。

HotRing 提供两种策略：

  1. 随机移动策略：反应快，但准确率较低；
  2. 统计采样策略：准确率更高，但反应稍慢。

#3.2.1 随机移动策略

随机移动策略是一个非常直接的方法。它不维护历史统计信息，而是周期性地根据某次即时访问决定是否移动头指针。

具体来说，每个线程维护一个线程本地计数器，记录自己处理的请求数。每执行 R 次请求，就判断一次是否触发头指针移动：

  • 如果第 R 次访问的是当前头指针所指的“热项”，则保持不动；
  • 否则，如果第 R 次访问命中了某个“冷项”，就把头指针移到该条目，使其成为新的热项。

参数 R 影响两方面：

  • R 小：反应更快，但头指针可能频繁无效移动；
  • R 大：移动更少，但热点变化响应更慢。

作者在实验中默认取 R=5，认为这能在低延迟与低额外开销之间取得平衡。

不过，这种方法也有明显局限：当负载偏斜不够明显、或者一个环中存在多个热点时，头指针可能频繁在多个热点之间来回摆动，造成额外开销而难以获得稳定收益。

#3.2.2 统计采样策略

为获得更高性能，HotRing 进一步设计了统计采样策略，它通过更准确地识别热点来换取略高的反应延迟。

#索引格式

现代机器的物理地址通常只用到 48 位，因此在 64 位 CAS 原子操作中还剩约 16 位可用于元数据。HotRing 利用这些额外位来存放统计信息。

  • 头指针由三部分组成：

    • Active 位：标记当前是否处于采样状态；
    • Total Counter（15 位）：记录该环总共采样到的访问次数；
    • Address（48 位）：指向当前头项地址。

  • 条目结构中包含：

    • Rehash 位：控制 rehash；
    • Occupied 位：保证并发修改正确性；
    • 剩余 14 位：记录该条目的访问次数。

这样，HotRing 可以在不额外增加空间成本的情况下，在环级别与项级别记录统计信息。

#统计采样过程

在大规模哈希表中（例如包含 2^27 到 2^30 个桶），持续更新所有环的统计信息会引起巨大开销，因此 HotRing 采用周期性采样。

每个线程维护一个线程本地请求计数器。每处理 R 次请求，就判断是否要启动一轮新采样：

  • 如果第 R 次访问的是热项，说明当前热点判断仍然较准确，不需要采样；
  • 否则，说明热点可能已经迁移，于是将 Active 位置为 1，启动该环的一轮采样。

进入采样后，后续访问该环的请求会同时更新：

  • 头指针中的 Total Counter
  • 对应条目的访问计数器

采样需要额外 CAS 操作，会暂时降低访问效率，因此 HotRing 选择将样本数设置为“环内条目数”这一数量级，认为这已经足够推断热点分布。

#热点调整

采样结束后，最后一个参与该轮采样的线程负责完成热点调整。

  1. 首先，它通过 CAS 原子地清除 Active 位，确保只有一个线程进入后续计算阶段。
  2. 然后，计算环中每个条目的访问频率：n_k / N，其中 N 是总访问数，n_k 是第 k 个条目的访问数。
  3. 接着，对环中每个可能作为头指针的新位置 t 计算其“收益” W_t。这里的收益本质上表示：如果把头指针放在位置 t，则整个环的平均访问代价是多少。
  4. 选择使 W_t 最小的位置作为新的头指针。

这一点非常关键：最佳头指针位置未必就是最热项本身。当一个环中存在多个热点时，这种方法能找到一个全局更优的位置，避免头指针在多个热点之间频繁振荡。

最后，系统将头指针通过 CAS 移动到新位置，并清空采样计数，为下一轮采样做准备。

#面向写热点的 RCU 优化

对于小于等于 8 字节的值，HotRing 支持原地更新，此时读与写都可以视为访问该条目本身。

但对于更大的值，必须使用 RCU（read-copy-update）。这时更新一个条目会创建新版本，并修改前驱条目的 next 指针来指向新版本。因此，如果热点条目本身需要频繁 RCU 更新，真正频繁被写入的其实还有它的前驱条目——因为每次更新都要修改前驱指针。

基于这一观察，HotRing 对统计采样策略做了小修改：对于一次 RCU 更新，不增加被更新条目的计数，而是增加其前驱条目的计数。这样，头指针会更倾向于移动到写热点的前驱位置，使整个 RCU 更新路径更短。

#3.2.3 热点继承

如果头指针所指的热项被更新或删除，系统必须把头指针移动到别的条目。若随机移动，很可能把头指到冷项上，导致热点检测机制频繁触发，整体性能反而下降。

HotRing 的做法是利用已有的热点信息进行“热点继承”：

  • 如果环里只有一个条目，那么更新或删除时直接通过 CAS 修改头指针即可；
  • 如果环里有多个条目：
    • RCU 更新头项时：由于时间局部性，最新版本很可能马上再次被访问，因此把头指针移动到新版本；
    • 删除头项时：将头指针简单移动到下一个条目，这是一个直接且有效的策略。

#3.3 并发操作

由于 HotRing 允许头指针移动，因此其无锁设计要比普通无锁哈希更复杂，主要体现在两个方面：

  1. 头指针移动本身会与其他线程并发执行，因此必须防止头指针指向非法条目；
  2. 当删除或更新某个条目时，必须检查头指针是否正指向该条目。如果是，就要正确而智能地移动头指针。

HotRing 基于已有无锁链表工作，使用原子 CAS 操作保证两个线程不会同时修改同一个 next 指针；若多个线程竞争，只有一个成功，失败者需要重试。

#读操作

读操作只需在环上按第 3.1 节的规则扫描并匹配目标 key，不需要额外同步，因此是完全无锁的。

#插入操作

插入会创建一个新条目，并修改其前驱条目的 next 指针。若两个插入同时竞争同一个前驱指针，CAS 保证只有一个成功，另一个线程重试即可。

#更新操作

HotRing 针对不同值大小设计了两种更新方式：

  • 原地更新：适用于 8 字节值，不影响其他操作，CAS 即可保证正确性；
  • RCU 更新：适用于较长值，需要创建新条目，会与插入 / 删除 / 其他 RCU 更新产生并发冲突。

作者举了几个例子说明：如果一个线程正在对条目 B 做 RCU 更新，另一个线程又试图在 B 后面插入新条目 C，那么虽然两个线程修改的是不同指针，但若不加控制，C 可能会“挂”在一个已经不可达的旧版本 B 后面，造成逻辑错误。

为解决这类问题，HotRing 使用条目中的 Occupied 位：

  1. 在执行 RCU 更新前，先把待更新条目的 next 指针原子地标记为 occupied；
  2. 一旦这个位被设置，其他需要依赖该指针的插入 / 更新 / 删除操作就会失败并重试；
  3. 更新完成后，再用新版本替换旧版本，并清除相应占用标记。

#删除操作

删除本质上是把“指向待删条目的指针”改为“指向其后继”。因此删除期间必须保证待删条目的 next 指针不再被其他线程并发修改。这里同样使用 Occupied 位。

例如，若线程 A 正在删除某个条目，而线程 B 同时试图通过它完成某个 RCU 更新，则后者必须失败并重试，直到删除完成为止。

#头指针移动

头指针移动是 HotRing 特有操作，也要与更新、删除等普通操作正确协同。

主要有两类情况：

  1. 由于热点识别策略触发的头指针移动：

    • 在把头指针移到一个新条目前，先设置该条目的 Occupied 位，确保其不会在移动过程中被更新或删除。

  2. 由于更新 / 删除当前头项而触发的头指针移动：

    • 若更新头项，需要先占住新版本条目，再把头指针移过去；
    • 若删除头项，不仅要占住被删条目，还要占住它的下一个条目，否则头指针可能移动到一个已经被其他线程修改过的位置，造成失效引用。

#3.4 无锁 Rehash

随着插入不断发生，每个环中的冲突条目数会持续增长，导致单次访问需要遍历更多条目，从而严重拖慢 KVS。为此，HotRing 设计了一种无锁 rehash 机制，使系统在数据规模增长时能够灵活扩容。

传统 rehash 通常根据装载因子（即平均链长）触发，但这种做法没有考虑热点影响，因此并不适合 HotRing。HotRing 改为根据访问开销（也就是平均查找所需的内存访问次数）来决定是否触发 rehash，从而更适应热点项增长。

整个无锁 rehash 包含三个阶段：

#1）初始化（Initialization）

后台 rehash 线程先创建一个大小为原表两倍的新哈希表。新旧表之间通过共享 tag 的最高位来划分数据：

  • 原哈希使用 k 位做桶索引；
  • 新哈希使用 k+1 位。

对于旧表中的一个桶，在新表中会对应两个桶头指针，各自管理一半 tag 范围。同时，rehash 线程还会创建一个特殊的 rehash node，其中包含两个子 rehash 条目，分别对应两个新桶头。普通数据项与 rehash 条目拥有相同结构，只是后者不存实际 KV 数据，并通过 Rehash 位加以标识。

#2）拆分（Split）

在这一阶段，rehash 线程把两个 rehash 条目插入原来的环中，把原环逻辑上切成两段，对应新表中的两个桶。

一旦插入完成，新表即被激活：

  • 新到来的访问从新表出发；
  • 尚未结束的旧访问仍然可以通过旧表与 rehash node 正确遍历数据。

从这一刻起，数据在逻辑上已经被分成两条访问路径，单次查找最多只需要扫描原来约一半的环长度。

#3）删除（Deletion）

最后，rehash 线程需要等待一个过渡期，确保所有从旧表发起的访问都已结束，这类似于 RCU 中的 grace period。等确认旧访问全部完成后，就可以安全删除旧表及 rehash 节点。

值得注意的是，这个过渡期只会阻塞后台 rehash 线程，不会阻塞前台访问线程。

#4. 评估

本节使用基于真实工作负载特征的基准测试，对 HotRing 的性能进行评估，并与无锁链式哈希及其他基线系统做对比。

#4.1 实验设置

#硬件环境

实验在一台双路 Intel Xeon E5-2682 v4（2.50GHz）机器上运行：

  • 2 个 CPU socket
  • 每个 socket 16 核，共 64 个超线程
  • 内存容量 256GB
  • 操作系统为 CentOS 7.4，Linux 3.10 内核

为获得更好性能，实验中将线程绑定到固定核心。

#工作负载

实验使用 YCSB 的核心工作负载，除了包含 scan 的 workload E 之外。每个 key-value 对中：

  • key 大小为 8 字节
  • value 大小分别取：
    • 8 字节：用于原地更新
    • 100 字节：用于 RCU 更新

每次测试固定装载 2.5 亿个 key。通过改变“key 数 / bucket 数”比值控制平均冲突链长度。作者还调节 YCSB 中的 Zipf 分布参数 q 生成不同偏斜程度的工作负载，以模拟日常与极端热点场景。

根据生产环境统计：

  • 日常场景下，q 大致在 [0.9, 0.99]
  • 极端场景下，q 大致在 [1, 1.22]

因此论文中选取 q = 0.99 与 q = 1.22 作为代表值。

#对比基线

为了体现热点感知设计的优势，作者实现了一个基于无锁冲突链的哈希索引作为基线（Chaining Hash），它修改自 Memcached 中的哈希结构，利用 CAS 把新条目插入到链头。

此外，还比较了以下系统：

  • FASTER：微软的高性能并发 KV 存储
  • Masstree：高性能内存范围索引，代表非哈希型 KVS
  • Memcached：传统加锁 KVS，实现上作为参考

为了公平，实验中严格控制各个方案的索引内存占用一致。

如果没有特别说明，默认实验配置为：

  • 64 线程
  • value 大小 8 字节
  • YCSB workload B
  • q = 1.22
  • key/bucket 比例为 8（对 HotRing）

#4.2 与现有系统的比较

#总体性能

实验结果表明，在各类 YCSB 工作负载下，HotRing 的吞吐量整体优于所有对比系统。

作者实现了两个变体：

  • HotRing-r：使用随机移动策略
  • HotRing-s：使用统计采样策略

在 workload B 和 C 等热点明显的场景下，HotRing-s 的优势尤为显著：

  • 单线程下达到 12.90M ops/sec
  • 64 线程下达到 565.36M ops/sec
  • 相比其他系统提升约 2.10× 到 7.75×

在包含大量插入的 workload D 和 F 中，HotRing-s 也优于其他方案，提升范围约为 1.81× 到 6.46×。原因是：

  • 有序环支持更快地提前终止定位；
  • tag 字段降低了排序与比较成本。

虽然 HotRing-r 的热点识别不如 HotRing-s 准确（大约差 7%），但整体性能仍显著好于其他系统。

#冲突链长度的影响

作者通过把 key/bucket 比从 2 调到 16，观察冲突长度增长时各系统吞吐量变化。

当 key/bucket 比为 2 时，Chaining Hash 和 FASTER 表现尚可，因为链较短，热点项额外内存访问成本不大，CPU Cache 的作用更明显。

但随着冲突链变长，Chaining Hash 与 FASTER 的性能明显下降，因为热点项仍然可能出现在较深位置，访问代价高。相比之下，HotRing 由于把热点项尽量放在头指针附近，即使链很长，性能仍保持较高水平。

例如：

  • 当 key/bucket = 2 时，HotRing 的吞吐量分别是 Chaining Hash 与 FASTER 的 1.93× 和 1.31×；
  • 当 key/bucket = 16 时，差距扩大到 4.02× 和 3.91×。

这说明 HotRing 的热点感知设计不仅提高了性能，也提高了可扩展性。

#访问偏斜程度的影响

作者把 Zipf 参数 q 从 0.5 提升到 1.22，以模拟热点不断加剧的过程。

结果显示：

  • Chaining Hash 与 FASTER 随着偏斜加剧，吞吐量提升并不明显，因为它们没有真正利用热点信息；
  • HotRing 的性能则随着 q 增大显著提升，尤其在 q > 0.8 时更明显。

即便在热点并不特别严重的情况下（q 在 0.5 到 0.8 之间），HotRing-s 仍然优于其他方案。这是因为统计采样策略能够处理一个环中存在多个热点的情况，找到更优的头指针位置；而 HotRing-r 在这种场景下更容易发生频繁移动，因而效果较差。

#RCU 操作的表现

为了突出 RCU 场景，作者使用 100 字节 value 并构造写密集工作负载（50% 写 / 100% 写）。

结果表明：

  • 若统计采样策略不对 RCU 特殊处理（记作 HotRing-s(w/o)），性能会非常差，甚至比 Chaining Hash 和 FASTER 更差；
  • 而采用了“为前驱计数”优化的 HotRing-s，则显著提升了 RCU 性能。

在 key/bucket = 2 时，HotRing-s 略慢于 FASTER，原因是：

  • 热点值所在位置距离头指针仍有一次额外访问；
  • RCU 更新还需要一次额外 CAS 来设置 Occupied 位。

但随着链变长，这些问题影响减弱。当 key/bucket = 8 时，HotRing-s 的吞吐量约为 Chaining Hash 与 FASTER 的 1.32×。

#4.3 关键设计的细节分析

#开销拆解

作者进一步分解了单次读访问的时间开销，包括：

  • 访问头指针的成本
  • 访问头项的成本
  • 访问非头项的成本
  • 计算哈希值的成本
  • 基准框架读写命令的成本
  • 其他系统开销

结果显示，在 Chaining Hash 中，最大的成本来自非头项访问。这意味着热点项常常分布在链中较深位置，导致每次读取都要多次跳转。

而在 HotRing-r 和 HotRing-s 中，非头项访问成本明显下降。尤其 HotRing-s 在高偏斜场景下，这部分成本极低，说明更多热点项被准确识别并放到了头部附近。

#反应延迟

热点识别策略的一个重要指标是：热点发生变化后，系统要多久才能恢复到稳定性能。

实验显示：

  • HotRing-r 反应最快，通常不到 2 秒就能稳定；
  • HotRing-s 反应稍慢，但峰值吞吐更高，因为其热点识别更准确；
  • Chaining Hash 不受影响，因为它没有热点感知能力。

这体现了一个典型权衡：随机策略偏快，统计策略偏准。

#读未命中（read miss）

作者比较了读未命中时的吞吐量。结果表明，随着链长度增加，HotRing 相比 Chaining Hash 的优势会越来越大。

例如：

  • key/bucket = 2 时，HotRing 提升约 1.17×；
  • key/bucket = 16 时，提升约 1.80×。

原因在于 HotRing 平均只需比较约半个环即可得出“未命中”结论，而传统链式哈希必须扫描整条链。

#参数 R 的影响

参数 R 控制头指针移动 / 启动采样的频率。实验表明：

  • R 太小：会因热点识别和头指针移动过于频繁而带来额外开销；
  • R 太大：则会因为热点变化响应过慢而损失性能。

综合来看，R = 5 是一个较好的折中点。

#尾延迟

HotRing-s 需要采样统计，且采样结束时最后一个线程还需额外计算最优头指针位置，因此可能产生长尾访问。

实验统计了 10 万次访问的延迟分布：

  • 当 q = 1.22 时，99 分位响应时间约为 2 微秒，但长尾可达 8.8 微秒；
  • 当 q = 0.99 时，99 分位约为 3 微秒，长尾约为 9.6 微秒。

作者指出，这部分长尾与实现中某些简化有关，未来可通过把额外计算挪到后台线程中进一步减轻。

#无锁 Rehash 效果

作者还构造了一个“50% 读 + 50% 插入”的场景来验证无锁 rehash：初始加载 2.5 亿个 key，key/bucket = 8，然后持续插入使哈希表不断增长。

结果表明，两次连续 rehash 能够在数据规模持续增大的过程中维持整体吞吐量稳定。rehash 期间虽然会出现短暂性能下降，但这是因为新哈希表刚开始工作时还未充分建立热点感知状态。整体而言，这种无锁 rehash 能够在不阻塞访问线程的前提下，较平滑地完成扩容。

#5. 相关工作

许多已有工作都在研究 KVS 的索引结构设计。

  • Memcached 是最广泛使用的分布式 KVS 之一，但由于多线程环境下锁竞争频繁，其并发性能并不理想。
  • 在 Memcached 基础上，后续大量工作通过引入无锁设计和 cache-friendly 优化，实现了更高并发度和吞吐量，例如 MemC3、MICA、FASTER 等。
  • 其中 FASTER 是当时代表性的高性能无锁 KVS 实现。
  • 在“热点感知”方面，Splay Tree 是一个很有启发性的经典工作，它会根据最近访问情况自适应地调整树结构；但其基于锁的设计不适用于高并发热点场景。

此外，还有大量工作关注如何结合新硬件提升 KVS 性能，例如：

  • FPGA 加速
  • RDMA 网卡
  • GPU
  • 用户态 TCP 栈
  • 基于 InfiniBand 的硬件级可靠通信

还有一些工作研究了高效无锁内存管理与 ABA 问题规避方法。作者指出，这些硬件或内存管理优化与索引结构设计是正交的，因此未来同样可以与 HotRing 结合，进一步提升性能。

#6. 结论与未来工作

在现实部署中，KVS 热点问题非常常见，而且近年来愈发严重。例如，为了保证高可用，阿里巴巴的 NoSQL 产品 Tair 不得不为突发热点预留超出正常需求的机器资源。因此，设计热点感知的内存 KVS具有很强现实意义。

基于对机会与挑战的分析，本文提出了一种名为 HotRing 的哈希索引，专门针对“大量并发请求集中访问少量条目”的场景进行优化。它通过把桶头指针动态移动到高频访问条目附近，适应热点迁移。在大多数情况下，热点项只需 两次内存访问 就能被取到。与此同时，HotRing 对常规哈希操作和其特有操作都采用了无锁结构。

大量实验表明，在高度偏斜负载下，HotRing 相比其他内存 KVS 可实现 2.58× 的吞吐提升。目前，HotRing 已经成为 Tair 的一个子组件，并在阿里巴巴集团内部得到广泛使用。

作者还指出：

  • HotRing-r 主要针对每条链只有一个热点的情况；
  • HotRing-s 则还能处理多个热点。

多数情况下，多热点问题可以通过 rehash 缩短链长来缓解。但在一些极端情形下，如何更好处理多热点，仍然是未来工作方向。

#论文核心思路速记

如果只看最核心的几点，这篇论文的 idea 可以概括为：

  1. 问题识别：内存 KVS 自己内部也有热点问题，而且越来越严重。
  2. 关键洞察：热点项不应像冷数据那样“平均对待”，应让它们更靠近桶头。
  3. 结构设计：把传统链式哈希改造成有序环，从而允许桶头指针自由移动。
  4. 热点适配：通过随机移动或统计采样，在运行时识别热点迁移。
  5. 并发实现：使用 CAS + Occupied bit + 无锁设计，保证头指针移动、更新、删除、rehash 全流程正确。
  6. 实际效果：在高度偏斜负载下，显著提高读写吞吐，尤其适合“少量 key 承受巨大访问压力”的场景。
